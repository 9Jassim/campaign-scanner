## Key Learnings

- **Project:** campaign-scanner — multi-store WhatsApp campaign scanner. Data isolated by `store_id`. Full spec in `PROJECT_BRIEF.md`.
- **Prisma 7 config:** Connection URLs live in `prisma.config.ts` (via `defineConfig` + `env()` from `prisma/config`), NOT in `schema.prisma`'s datasource block (`url`/`directUrl` were removed). Runtime `PrismaClient` requires a driver adapter — we use `@prisma/adapter-neon` in `lib/db.ts`.
- **Prisma 7 CLI:** `migrate diff` flag `--to-schema-datamodel` was renamed to `--to-schema`. The dotenv banner leaks to stdout, so redirecting `migrate diff --script > file.sql` pollutes line 1 — strip it.
- **Migrations offline:** Generated the initial migration without a live DB using `prisma migrate diff --from-empty --to-schema prisma/schema.prisma --script`, plus a hand-written `migration_lock.toml`. Pipe through `grep -v '◇ injected env'` to strip the dotenv banner from stdout.
- **NextAuth v5 structure:** Split config into `auth.config.ts` (edge-safe, no DB/bcrypt, has the `authorized` route-protection callback) used by `middleware.ts`, and `auth.ts` (Node runtime, Credentials provider with Prisma+bcrypt). Credentials login requires `session.strategy: 'jwt'` (no DB sessions). Middleware matcher MUST exclude `/api/auth` or login POSTs get redirected. `user_profiles` gained `password_hash` + unique `email` + `@default(cuid())` id.
- **Portal pages pattern:** List pages (`/contacts`, `/receipts`, `/raffle`) are server components gated by `requireManager()` (cashiers → `/scanner`); `/admin/*` uses `requireAdmin()`. Store scoping via `resolveStoreScope(profile, searchParams.storeId)` → `{ stores, selectedStoreId, scopeIds }`; every query filters `storeId: { in: scopeIds }`. Filters use a GET `<form>` (state in URL, stays server-rendered). Settings + admin-users mutate via co-located `'use server'` action files that redirect back with `?saved=1`/`?error=`. Store assignment = `store_users` rows, edited via checkboxes in `/admin/users`. Meta access token encrypted with `lib/crypto.ts` (AES-256-GCM, `ENCRYPTION_KEY` base64 32 bytes).
- **Verified end-to-end (browser + live Neon):** login → scan (SI-100008 → 4 entries) → data appears in receipts/raffle → store filter isolates → created a cashier scoped to one store. All green.
- **Per-store list pages:** Contacts/Receipts/Raffle show ONE store at a time (no combined view). `resolveActiveStore(profile, storeId)` defaults to first accessible store; `FilterBar allowAllStores={false}` hides the "All stores" option. Store name shown in the page heading.
- **Auto-submit dropdowns:** `components/auto-submit-select.tsx` (client) calls `form.requestSubmit()` on change — used for store filter, receipts status filter, settings store switcher. Text search still uses Apply/Enter.
- **WhatsApp send:** `lib/whatsapp.ts` — `hasWhatsAppCredentials(store)`, pure `buildTemplatePayload(store, params)` (exported so the 12 positional vars are unit-testable; Meta matches {{1}}..{{12}} by ARRAY ORDER, so order is load-bearing), and `sendWhatsApp()` which never throws (returns `{wamid|error|rateLimited|skipped}`). Token decrypted per-call from `stores.metaAccessTokenEncrypted`. `to` must be digits-only (no `+`). 15s timeout via `AbortSignal.timeout`. Rate limit = HTTP 429 or codes {4, 80007, 130429, 131048, 131056} → receipt stays `pending` + row added to `retry_queue` (retry cron itself NOT built yet). In `lib/scan.ts`, the send happens AFTER the Serializable txn commits (never hold a txn open across a network call, never fail a committed scan); receipt starts `pending` (or `skipped` with no creds) then updates to sent/failed/pending.
- **Webhook (`/api/webhook`, public — excluded from auth in auth.config.ts):** GET echoes `hub.challenge` as text/plain when `hub.verify_token` === `WEBHOOK_VERIFY_TOKEN`, else 403. POST reads the RAW body first (signature is over exact bytes) and ALWAYS verifies `X-Hub-Signature-256` against the addressed store's secret — see the per-store webhook entry below; the old "optional when `META_APP_SECRET` is set" behaviour was a fail-open and is gone. Status logic lives in pure `lib/webhook.ts` `resolveStatusUpdate(current, event)`: MONOTONIC ranking (skipped/pending 0 < sent 1 < delivered 2 < read 3) so out-of-order Meta events never regress; `failed` always applies with formatted error; returns null = no-op (duplicate/stale). Statuses match receipts by `wamid` **scoped to the signing store** (never by wamid alone — that let one store rewrite the other's receipts); inbound messages dedupe on wamid and are attributed to the signing store. Returns 500 on unexpected errors so Meta retries (writes are idempotent), 200 on malformed JSON (retry would never help).
- **Amounts: ALWAYS parse with `parseAmount()` from `lib/barcode.ts`, never `parseFloat`.** The rule is "the LAST separator is the decimal point, everything before it is grouping" — so `1,200.00`, `1.200,00` and `12,345.67` all read correctly, and `45,500` is 45.5 BD (Bahraini fils). The scanner used `parseFloat(s.replace(',', '.'))`, which only swaps the FIRST separator, so a hand-typed `1,200.00` became `1.200.00` → 1.2 BD → 0 entries instead of 120, silently. The user's POS always prints decimals (`15.00`), so the one genuinely ambiguous shape — a whole number with grouping and no decimals, `1,200` — never occurs in practice; it would still read as 1.2 BD. The scanner now warns (amber, `role=alert`) whenever an amount is present but earns 0 entries, so a below-minimum receipt is loud rather than just a greyed-out button.
- **Theme is `data-theme` on `<html>`, NOT prefers-color-scheme:** `globals.css` redirects Tailwind's variant with `@custom-variant dark (&:where([data-theme="dark"], [data-theme="dark"] *))` — without that, every `dark:` utility in the app keeps following the OS and disagrees with the toggle. localStorage (`campaign-scanner-theme`) holds the *preference* (light|dark|system); `data-theme` holds the *resolved* value and is always concrete. A blocking inline script in `<head>` (`components/theme-script.tsx`) applies it pre-paint; `<html suppressHydrationWarning>` is required because of it. `ThemeToggle` starts as 'system' on both server and client and reads localStorage in an effect, so no button flashes as active. Toggle lives in the nav and on the sign-in page (no nav there).
- **THE PRIZE DRAW IS RUN OFF THE GOOGLE SHEET (user, 2026-07-19).** This makes the sheet's completeness a correctness requirement, not a convenience: any row missing from the Raffle tab is an entry that cannot win. Treat a failed/partial/stale sync as a serious bug, and never add a code path that silently writes fewer rows than the database holds.
- **Built for scale — the campaign expects large raffle volumes (6 months, 2 stores):** Sheets writes are batched at 10k rows/request (`ROWS_PER_REQUEST`) because a single request dies past ~50k rows (measured: 50k=4.3MB, 100k=8.6MB, 300k=25.9MB). The CSV export streams via `ReadableStream` with **id-cursor** paging (`PAGE_SIZE` 5000) and has **no row cap** — it previously `take:`-ed 10,000 and silently dropped the rest. Every export `orderBy` ends with `{ id: 'asc' }` so the ordering is total; without that, rows sharing a timestamp can shuffle between pages and be duplicated or skipped. Pagination has first/last buttons and a page-number box because a big raffle runs to thousands of pages.
- **Google Sheet = DAILY readable view + their weekly archive (user's decision, 2026-07-16):** The brief specifies a per-scan dual-write; the user overrode that twice — first "what we have in the sheet is the weekly backup", then "let's make the portal update the sheet every day so we can view it. and we will keep the sheet to backup every week". So there is deliberately **no dual-write**: a nightly Vercel Cron (`vercel.json`, `0 21 * * *` = 00:00 +3) hits `/api/cron/sheets-sync`, which replaces each store's `Contacts`/`Log`/`Raffle` tabs with a FULL snapshot; their Apps Script archives the sheet weekly (Sunday `atHour(2)`, script TZ Asia/Qatar = +3, fires within 02:00–03:00). The nightly sync lands ahead of it by design. Nothing is ever read back OUT of the sheet — the portal is the source of truth. Orchestration lives in `lib/sheets-sync.ts` (db + network), shared by the cron and `/admin/sheets`; row building stays in pure, tested `lib/backup.ts`. `stores.last_sync_at/_status/_detail` (migration `0004`) drive the admin page. Admins get **Sync now** and **Sync now (force)** — force bypasses the never-shrink guard, needed after deliberately clearing data. This kills the brief's awkward "Sheets has no upsert" problem entirely. Because snapshots are complete (not deltas), the sheet's own Apps Script archive always captures a valid copy — ordering the two jobs only affects *freshness*, never completeness, so schedule the sync ~1h BEFORE their script (syncing after makes every archive a week stale). `isSafeToOverwrite` refuses a snapshot smaller than the sheet, so a half-built sync can't destroy rows. `lib/google-sheets.ts` hand-rolls the service-account JWT (RS256 via node:crypto) rather than pulling in `googleapis` (huge; cold starts); `overwriteTab` writes THEN trims — never clear-then-write, which would leave the tab empty if the write failed. Cells are prefixed with `'` against formula injection (cashier-typed names land in a spreadsheet). Route is public in auth.config.ts and authenticates with `CRON_SECRET` (unset = refuse, never run open).
- **Scan concurrency — per-store advisory lock (`lib/store-lock.ts` + `lib/scan.ts`):** Each scan takes `pg_advisory_xact_lock(storeLockKey(store.id))` as the FIRST statement in its transaction; same-store scans queue, different stores never block (verified: 10 scans across 2 stores took 1.08x the time of 5 in one store). Key = `sha256(storeId).readBigInt64BE(0)` — advisory locks need a signed int64, store ids are UUIDs. **The transaction MUST be `ReadCommitted`, never Serializable/RepeatableRead** — those freeze the snapshot at the first statement (the lock itself), so a waiter acquires the lock and *still* reads pre-commit data, making `MAX(entry_number)+1` stale and the lock useless (proven empirically). **Take the lock with `$executeRaw`, NOT `$queryRaw`** — `pg_advisory_xact_lock()` returns `void` and the Neon adapter throws `UnsupportedNativeDataType` trying to deserialize it. `SET LOCAL statement_timeout = '10s'` bounds the wait → SQLSTATE 57014 → `ScanError('System busy, please try again', 'busy')` → HTTP 503; the Prisma txn `timeout` must exceed it (15s) or Prisma tears down first. WhatsApp send stays OUTSIDE the txn — holding the store lock across a network call would stall every till in that shop.
- **Webhook is per-store authenticated, and fails CLOSED:** One URL serves two legally-separate companies, so a request is not "signed or not" — it is signed by exactly ONE store's app and may only touch that store's rows. Flow: parse body → `phoneNumberIdsIn(payload)` (safe to read unverified — it only picks WHICH secret to check; a lie fails verification) → resolve candidate stores by `metaPhoneNumberId` → `findSigner(raw, header, candidates)` returns the matching candidate (NOT a boolean — the identity is the whole point) → then TWO guards: any change whose `phone_number_id` != the signer's is refused as cross-store, and receipt lookups are scoped `{ wamid, storeId: signer.store.id }` (a wamid alone is not proof of ownership). Status codes: no candidate secret → **503** (our config; Meta retries so events can still land once fixed), signature mismatch → **403**, no phone_number_id → 200 + drop, processing error → 500 (writes are idempotent). **There is deliberately no "no secrets configured = skip verification" path** — that used to make a broken `ENCRYPTION_KEY` silently open the endpoint. A decrypt failure yields no candidate rather than skipping. **While two stores share the `META_APP_SECRET` env fallback they can still sign for each other — per-store App secrets in Settings are what make the isolation real.**
- **Login is by USERNAME, not email (migration `0005`):** Till staff rarely have a work email, so `user_profiles.username` (unique, lowercase, `/^[a-z0-9._-]{3,32}$/`) is the credential and `email` is optional contact detail (nullable + unique — Postgres allows many NULLs under a unique index). `receipts.cashier_username` records who rang a sale up, captured at scan time so it survives renames/deletion. Displays show `fullName ?? username`. Seed uses `SEED_ADMIN_USERNAME`. **Adding a NOT NULL unique column to a populated table fails** — the migration adds it nullable, backfills from the email local-part, then sets NOT NULL.
- **Sessions expire after 2h, sliding:** `auth.config.ts` sets `session.maxAge = 2*60*60` AND `updateAge = 5*60`. **`maxAge` alone does not slide** — NextAuth only rewrites the token once `updateAge` has elapsed, so without it an active cashier is logged out mid-shift at the 2h mark. The issued cookie carries `Expires`, NOT `Max-Age` — don't grep for the wrong one when verifying.
- **Sign-in throttling:** `login_attempts` table (migration `0003`) counts failures per key. `lib/login-throttle.ts` = PURE policy + `signInErrorMessage`/`lockoutCode` (no db import, so it's testable AND safe to import from the client form). `lib/login-attempts.ts` = the Postgres reads/writes + opportunistic GC. Account (`user:<username>`, `ACCOUNT_POLICY`): 5 failures → lock, doubling 1→2→4→8→15min cap. IP: 30 failures — deliberately loose, because a shop's cashiers share one NAT and locking the store mid-shift stops them scanning. Success clears the account key only (the IP key is shared; one valid login must not hand an attacker a fresh budget). `auth.ts` throws a `CredentialsSignin` subclass whose `code` carries the wait (`too_many_attempts:60`); NextAuth surfaces it as `signIn()`'s `res.code`, which the form renders. **Deploy order matters: the migration MUST be applied before the code ships, or every login throws on the missing table.**
- **Passwords (`lib/password.ts`):** `BCRYPT_COST` is defined ONCE here — never re-hardcode `bcrypt.hash(pw, 10)` (it was previously copy-pasted in auth.ts, admin/users/actions.ts and seed.ts). `verifyPassword(pw, hash|null)` falls back to `ABSENT_USER_HASH` for unknown emails so the bcrypt time still burns — otherwise an unknown email returns instantly (~0ms vs ~900ms) and the timing gap enumerates the staff list.
- **Dates are ALWAYS Bahrain time (`lib/datetime.ts`):** Timestamps are stored in clean UTC (Postgres `TimeZone=GMT`, columns are `timestamp without time zone`; verified JS-vs-PG drift 0.8s) — storage is correct, so date bugs here are display bugs, not data bugs. All list pages are SERVER components, so `toLocaleString()`/`toISOString()` follow the *server's* zone (UTC on Vercel) and render 3h early. Use `formatDateTime` (UI, `15/07/2026, 14:35`), `formatDateTimeCsv` (export, `2026-07-15 14:35:11`), `todayInBahrain` (filenames). Formatters are module-level `Intl.DateTimeFormat` with explicit `timeZone: 'Asia/Bahrain'` + `hourCycle: 'h23'` (plain `hour12: false` can render midnight as `24:00`). Bahrain is UTC+3 year-round, no DST.
- **Permissions + export:** Roles unchanged (admin = full control per user's call). Per-user `permissions` jsonb on `user_profiles` (migration `0001_add_permissions`, hand-written since Prisma 7 `migrate diff --from-migrations` needs a shadow DB). `canExport(profile)` in lib/auth (admin always true; others need `permissions.canExport`). Admin user form has a "Can export" checkbox. CSV export via `/api/export?type=contacts|receipts|raffle&storeId=&q=&status=` (honors filters + store scope, gated by canExport, UTF-8 BOM for Arabic). Export button on the 3 list pages, shown only when canExport.

## Do-Not-Repeat

- [2026-07-14] `create-next-app` defaults to Next 16 + React 19. After pinning to Next 14, three scaffold leftovers break the build: `next.config.ts` (Next 14 needs `.mjs`/`.js`), the flat `eslint.config.mjs` (use `.eslintrc.json` + eslint 8 + eslint-config-next 14), and `Geist`/`Geist_Mono` fonts in `layout.tsx` (not in Next 14's `next/font/google` — use `Inter`).
- [2026-07-16] Google service-account setup: `GOOGLE_SERVICE_ACCOUNT_PRIVATE_KEY` must be the JSON's **`private_key`** (~1700-char PEM starting `-----BEGIN PRIVATE KEY-----`), NOT **`private_key_id`** (40 hex chars) — the fields sit next to each other and the user pasted the wrong one into both .env.local and Vercel, giving `error:1E08010C:DECODER routines::unsupported` on every sync. Diagnose by SHAPE, never by printing the value: length, `-----BEGIN` prefix, `/^[0-9a-f]{40}$/`. Before running any sync that overwrites, do a READ-ONLY probe first (`buildServiceAccountJwt` + `tabRowCount` per tab) — it proves key + sharing + sheet id + tab names without writing a thing. In Vercel, env values take NO surrounding quotes (dotenv strips them in .env.local; the dashboard does not), but the `\n` escapes inside the key must stay literal.
- [2026-07-16] After deleting an App Router route directory, `npx tsc --noEmit` still fails on stale generated types in `.next/types/...` pointing at the removed file. `rm -rf .next` and re-run; it is not a real error.
- [2026-07-16] NEVER run `next build` while the dev server is running — they share `.next`, and the build clobbers the dev server's chunks mid-request (`Cannot find module './vendor-chunks/next.js'`, every route 500s). Stop the preview server first, or `rm -rf .next` and restart it afterwards.
- [2026-07-16] Prisma 7 renamed `migrate diff --from-schema-datasource` to `--from-config-datasource` (takes no path — it reads `prisma.config.ts`). This is the way to generate a delta migration WITHOUT a shadow DB: `npx prisma migrate diff --from-config-datasource --to-schema prisma/schema.prisma --script`. Beats hand-writing SQL, and an empty diff also proves the live DB has no drift.
- [2026-07-16] A user asked to fix security finding #2 (login rate limiting); I did the research, got interrupted by an unrelated bug report, pivoted, and never returned — then implied #2 was done and moved on to #1. When a request is interrupted, re-check the earlier one is actually finished before reporting or proposing next steps. `git log` + grep is the cheap way to confirm rather than trusting memory.
- [2026-07-16] Never format a date in a server component with bare `toLocaleString()`/`toISOString()` — it uses the SERVER's timezone (UTC on Vercel), so every invoice rendered 3 hours early in production. This class of bug CANNOT reproduce locally: the dev machine is `Asia/Bahrain`, so the broken code looked correct in dev. Always pass `timeZone` explicitly via `lib/datetime.ts`. Corollary for verification: rendering the page on the dev machine proves nothing about a timezone fix — re-run under `TZ=UTC` (e.g. `TZ=UTC npx vitest run`, or a `tsx` script importing the real formatter against real rows) to reproduce Vercel.
- [2026-07-16] `npx tsx foo.ts` fails on top-level await ("not supported with the cjs output format") because package.json has no `"type": "module"`. Name throwaway scripts `.mts` to force ESM. Also: scratch scripts must live INSIDE the project dir or `node_modules` resolution fails (dotenv/prisma not found).
- [2026-07-14] Browser automation on the scanner: the barcode input is React-controlled, so `form_input` sets the DOM value WITHOUT firing React's onChange — state stays empty and parsing/submit no-ops. Use the `type` action (real keystrokes) after clicking, and reset via the app's own "Clear" button, not `form_input`. Also: the key action name is `Enter` (not `Return`) to trigger onKeyDown. Uncontrolled server-action forms (admin users) are fine with `form_input`.
